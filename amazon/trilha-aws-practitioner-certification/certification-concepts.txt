Diferença entre SNS e SQS:
O 'SNS' é um sistema distribuído de publicação e assinatura . As mensagens são enviadas aos assinantes como e quando são enviadas pelos editores ao SNS.

O 'SQS' é um sistema de enfileiramento distribuído . As mensagens NÃO são enviadas aos destinatários. Os receptores precisam pesquisar ou receber mensagens do SQS . As mensagens não podem ser recebidas por vários receptores ao mesmo tempo. Qualquer receptor pode receber uma mensagem, processar e excluí-la. Outros receptores não recebem a mesma mensagem posteriormente. A pesquisa introduz inerentemente alguma latência na entrega de mensagens no SQS, diferentemente do SNS, onde as mensagens são imediatamente enviadas aos assinantes. O SNS suporta vários pontos finais, como email, sms, ponto final http e SQS. Se você deseja que um número e tipo de assinantes desconhecidos recebam mensagens, precisará do SNS.

Link: https://qastack.com.br/programming/13681213/what-is-the-difference-between-amazon-sns-and-amazon-sqs

> Identity and Access Management (IAM)
  USER GROUP ROLE [autenticação] POLICY-DOCS[autorização] [U.G.R.P]
  
  -User: Is a operator, cloud be a human or machine. The credencials are PERMANENT. 'USER' BELONGS TO 'GROUP'
  -Group: grupo de usuários 'GROUP' BELONGS TO 'USER'
  -Role: NOT rules/permisson. Is the AUTENTICATION methods. TEMPORARY. As roles ocorrem em um arquivo separado conhecido como policy docs.
  -Policy Docs [AUTORIZATION]: is a JSON document that could be vinculataded a user, group or role. Policy docs has a APIS groups, white list; Has black too list with explicit negations operations.
        Ex.: hour of the day, specific network, operations functions
		Pratice example: send file to S3. Checks user name, password and secret access key.
		Pratice example 2: encerrar/interromper a execução em produção. Cria um doc policy que nega (DENY) a parada do Ec2 e enceramento do Ec2 e aplicar isso para todos os USERS, GROUPS and ROLES.
		Pratice example 3: hacker has a account and password one user. The security management dont know which user, he just delete all POLICY DOCS.
		O hacker tenta apagar um bucket no S3, o sistema vai autenticar o user que o hacker tem (ROLE), mas quando o policy docs for autorizar vai NEGAR.
		A ação será registrada no 'AWS CLOUD TRIAL'.
	Obs: toda ação de API (sucesso ou não) será registrada no CLOUD TRIAL (log de segurança)

					A definição de preço varia de acordo com:

					Instâncias Spot

						Use o Elastic Load Balancing para distribuir o tráfego

						Ajusta automaticamente o número de instâncias

					Endereços IP elásticos

		Amazon EBS
			Armazenamento em bloco para instâncias
			Os volumes perssitema independentemente da instância
			Discos análogos à virtuais na nuvem
			Três tipos de volumes:
				Uso geral (SSD)
				IOPS provisionadas (SSD)
				Magnético (HD)
			Fatores de custo
				Volumes: todos os tipos de volume são cobrados pela quantidade provisionada por mês em GB
				IOPS:
					Uso geral (SSD): incluído no preço
					Magnético (HD): cobrado por número de solicitações E/S
					IOPS provisionadas (SSD): cobradas pela quantidade provisionada em IOPS
				Snapshots
					Uso adicional por GB/mês de dados armazenados
					Entrada não cobradas
					Saída de dados é cobrada

					Armazenamento de backup para instâncias de banco de dados encerradas cobrando a GB/mês
				Armazenamento adicional:
					Backup além do armazenamento provisionado cobrado a GB/mês
				Tipos de implantação:
					Cobranças de armazenamento e I/O
					Única Zona de Disponibilidade
					Múltiplas Zonas e Disponibilidade
				Transferência de dados:
					Não há cobrança pela entrada
					Há cobrança pela saída, pelo volume (faixas transferidas)
